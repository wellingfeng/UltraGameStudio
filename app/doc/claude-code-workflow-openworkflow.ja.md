# Claude Code に Dynamic Workflows が来た。では他のモデルは？ オープンソースの代替、OpenWorkflows

## 最近、Claude Code が新しく出した dynamic workflows を見ている。MCP、Skill、Hooks と比べると、この新機能について話している人はまだ少ない。以下ではまとめて workflows と呼ぶ。

複雑なタスクでは、以前はまず調査用の HTML を作り、それを技術方案の HTML に変換し、最後に AI に渡して開発させる人が多かった。ただ、結果があまり良くないことも多い。主な理由は、HTML は人間が読むためのテキストであって Script ではなく、構造化情報が不足しているからだ。順序の一貫性、並列化できる度合い、境界が明確かどうか、タスクをどう分けるか、タスク間で情報をどう交換するかが曖昧で、AI が推測する部分が多すぎる。

一方、workflows はそれ自体が Script なので、この問題を根本的に解決できる。

さらに workflows には、多角的な探索、敵対的な検証、方案への投票メカニズムもある。精度が高くなる理由もそこにある。規模で勝つ方式だ。同じ問題に 5 つの agents を同時に走らせ、最後に別の agent がまとめる。確かにより正確になるが、token も一気に消費する。

ここまで汎用的なら、なぜ特定のモデルや特定の CLI に縛られる必要があるのか？

その考えから OpenWorkflows を開発した。より正確には、AI が開発した。OpenWorkflows は Claude Code のような workflows をビジュアルキャンバスにし、同じ流れを Claude Code、Codex、Gemini、さらに他のローカルまたはクラウドのランタイムに向けられるようにしている。

今回は抽象的な概念の話はせず、スクリーンショットに沿ってそのまま見ていく。例も具体的だ。OpenWorkflows に複数のインターフェーススタイルを対応させ、デフォルトを Pencil にし、「設定 / 外観」で切り替えられるようにする。

開発中は、できるだけ OpenWorkflows の中で作業し、自分自身をブートストラップできるようにした。

以下の流れでは、開発用のデフォルト大規模モデルとして CodeX を使う。

### 0. まず最終画面を見る

<p align="center">
  <img src="images/0-标题使用.png" alt="OpenWorkflows のメインインターフェース" width="960">
</p>

OpenWorkflows のメイン画面では、中央に workflows のブループリント、右側にノードプロパティ、下部に AI 入力と返答がある。

メイン画面は大きく 4 つに分かれる。左が workflows 履歴、中央がビジュアルキャンバス、右がノードプロパティとよく使うプロンプト、下部が AI 入力と応答だ。

### 1. OpenWorkflows をダウンロードする

<p align="center">
  <img src="images/1-下载.png" alt="OpenWorkflows GitHub Releases" width="840">
</p>

GitHub プロジェクトページ右側の Releases から最新版を見つける。

### 2. 先に大規模モデルを設定する

デフォルトでは、システムに設定済みの CLI を使って起動する。CC-Switch などで設定できる。

### 3. 新しい workflows を作成し、要望を入力する

<p align="center">
  <img src="images/3-新建workflow.png" alt="新しい workflows を作成して要望を入力する" width="840">
</p>

モデルを設定したら、左側の「新しい workflows」をクリックする。キャンバスには Start、1 つの Agent、End という最小構成が表示される。

実際にノードを手で描く必要はない。右下の AI 入力欄を使う。この例では次のように入力した。

```text
OpenWorkflows で複数のインターフェーススタイルをサポートし、
デフォルトは Pencil デザインにして、
「設定 / 外観」で切り替えられるようにしたい。
```

書き終えたら Ctrl+Enter を押すか、右下の送信ボタンをクリックする。OpenWorkflows はこの自然言語を、編集可能な workflows ブループリントに変換する。

### 4-1. workflows ブループリントを生成する

<p align="center">
  <img src="images/4-1生成workflow蓝图.png" alt="生成された workflows ブループリント" width="960">
</p>

要望を送信すると、OpenWorkflows はまず現在のステップを完全な workflow に組み替える。

スクリーンショットのブループリントは、おおよそ次のような形だ。

```text
Start
  -> 外観サポート方案を並列で整理
      -> 既存の外観入口を調査
      -> マルチスタイル体系を設計
      -> Pencil デフォルトスタイルを設計
  -> 実装方案をまとめる
  -> 複数インターフェーススタイルを実装
  -> 設定 / 外観の切り替えを接続
  -> 検証と回帰チェック
  -> 交付結果を記録
  -> End
```

右側のノードプロパティでは、選択中ノードの属性を引き続き変更できる。もちろん多くの場合は、下部の入力欄から AI にブループリントノードを変更させ、継続的に反復する。

### 4-2. 生成されたスクリプトを見る

<p align="center">
  <img src="images/4-2蓝图脚本.png" alt="生成された workflows スクリプト" width="960">
</p>

上部には「スクリプト」入口がある。開くと、現在のブループリントから生成されたスクリプトが表示される。

スクリーンショットでは parallel(...) や agent(...) のような構造が見える。並列ノードは同時実行される分岐になり、通常ノードは個別の agent 呼び出しになる。

ここからも、OpenWorkflows が単に図を描くだけではないことが分かる。キャンバスの背後には統一された workflows 構造があり、それがあるから後で別のランタイムに接続できる。

### 5. 右側のよく使うプロンプトでさらに変更する

<p align="center">
  <img src="images/5-使用常用提示词.png" alt="よく使うプロンプトで workflows を続けて編集する" width="960">
</p>

ブループリント生成後、すぐに実行する必要はない。右側の「よく使うプロンプト」は、流れをさらに磨くのに向いている。もちろん自分で手書きしてもよい。

プロンプトはシーンごとに分類されている。たとえば対話的な確認、明確さ、完全性、コスト、構造、信頼性、性能と並列性、検証とテストなどだ。

スクリーンショットで押しているのは「要件を明確化」だ。AI がブループリントを変更する前に、重要な曖昧点を対話的に確認するよう、AI 入力欄にプロンプトを入れる。

これは実用的な設計だ。多くの workflows が失敗するのはモデルができないからではなく、最初に目標、境界、失敗パス、コスト戦略が明確でないからだ。

ほかにも grill-me、境界条件の補完、並列最適化、単一原則などのよく使うプロンプトがある。自分で追加したり変更したりもできる。

### 6. 対話選択で境界を確認する

<p align="center">
  <img src="images/6-交互选择.png" alt="対話選択で境界を確認する" width="640">
</p>

「要件を明確化」を押すと、AI はすぐに図を変更せず、まず「インターフェーススタイル切り替え機能をどの範囲まで実装するか？」と尋ねる。

スクリーンショットでは 2 つの選択肢がある。Pencil デフォルトスタイルだけを実装して拡張構造を残すか、Pencil と複数の切り替え可能なスタイルを同時に実装するかだ。

選択すると、AI はその決定を workflows ブループリントに書き戻し、更新後の IRGraph を出力する。このステップによって、AI が勝手に間違った方向へ進む問題を減らせる。

### 7. 実行をクリックする

<p align="center">
  <img src="images/7-运行.png" alt="workflows を実行する" width="960">
</p>

ブループリント構造、モデル設定、重要な境界が確認できたら、上部の「実行」をクリックする。

生成直後にすぐ実行するのはおすすめしない。まず並列分岐が妥当か、集約ノードが並列分岐の後にあるか、検証ノードが最終結果をカバーしているかを確認する。

あるノードの責務だけが曖昧なら、先にノードプロパティで変更してから実行すればよい。

### 8. 実行状態を見る

<p align="center">
  <img src="images/8-运行中.png" alt="workflows の実行状態を見る" width="960">
</p>

実行後、上部ボタンは「実行中…停止」に変わる。下部の AI 入力はロックされ、実行中にブループリントが乱れないようになる。

キャンバスにはノードの状態が表示される。スクリーンショットでは Start が完了し、後続の並列ノードが実行中で、右上には実行カウントもある。途中で失敗しても、前のタスクから続行できる。

### 9. インターフェーススタイルを切り替える

<p align="center">
  <img src="images/9-切换风格.png" alt="インターフェーススタイルを切り替える" width="840">
</p>

OpenWorkflows の開発が完了したら、プログラムを再起動し、「設定 / 外観」で異なる外観スタイルを切り替える。

スクリーンショットでは Pencil、Deep Night、Aurora、Daylight、Ember などのスタイルカードが見える。選択すると、グローバル背景、パネル、境界線、実行状態の色に反映される。

### 本当に役に立つと思うところ

OpenWorkflows の価値は、prompt に UI を被せることではない。

「要望 -> ブループリント -> スクリプト -> 実行 -> 履歴確認」をつなげている。自然言語で流れを生成し、キャンバス上で構造を確認し、必要ならよく使うプロンプトで境界を補い、最後に実行できる。

同じ workflows が 1 つのモデルに自然に縛られる必要もない。簡単なノードは安いモデルを使い、重要なノードは強いモデルを使い、実行先も Claude Code、Codex、Gemini、その他のランタイムへ広げられる。

複雑な AI コーディングタスクでは、この分解のほうが巨大な prompt 1 つより保守しやすい。あるノードが失敗したらそのノードを直す。不要な分岐があれば削除する。再利用したければ履歴から続ける。

### まだ早いが、方向性は見る価値がある

workflows という概念全体はまだ初期段階で、OpenWorkflows 自体も始まったばかりだ。ランタイム適配、ノード能力、スクリプトエコシステムはこれからも変わっていく。

ただし方向性ははっきりしている。AI コーディングは「チャット欄を開いて、毎ステップ手で進める」状態に長く留まらない。

複雑なタスクは最終的に workflows になる。見える、編集できる、移行できる、再利用できるからだ。

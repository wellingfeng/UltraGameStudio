# У Claude Code появились Dynamic Workflows. А что с другими моделями? Open-source альтернатива: OpenWorkflows

## В последнее время я смотрю на новые dynamic workflows в Claude Code. По сравнению с MCP, Skill и Hooks эту новую функцию обсуждают мало. Дальше я буду называть их просто workflows.

Для сложных задач многие раньше сначала делали HTML с исследованием, потом превращали его в HTML с техническим планом, а затем отдавали ИИ на разработку. Но результат часто был не очень. Главная причина в том, что HTML — это текст для человека. Это не Script, и ему не хватает структурированной информации. Последовательность, степень параллельности, четкость границ, разбиение задач и обмен информацией между задачами остаются неясными, поэтому ИИ приходится слишком много угадывать.

А workflows сами по себе являются Script, поэтому они могут решить эту проблему напрямую.

Кроме того, workflows дают исследование с разных углов, adversarial validation и механизм голосования за планы. Поэтому точность становится выше. Они побеждают масштабом: пять agents одновременно работают над одной проблемой, а затем еще один agent сводит результаты. Это действительно точнее, но tokens тоже сгорают очень быстро.

Раз это настолько универсально, почему оно должно быть привязано к одной модели или одной CLI?

Следуя этой мысли, я разработал OpenWorkflows, точнее, его разработал ИИ. Он превращает workflows в стиле Claude Code в визуальный холст и пытается сделать так, чтобы один и тот же поток мог быть нацелен на Claude Code, Codex, Gemini и другие локальные или облачные runtimes.

В этот раз я не буду говорить об абстрактных концепциях. Просто пройду по скриншотам. Пример конкретный: сделать так, чтобы OpenWorkflows поддерживал несколько стилей интерфейса, по умолчанию использовал Pencil и позволял переключаться в Настройки / Внешний вид.

Во время разработки я старался по возможности работать внутри OpenWorkflows, чтобы он мог bootstrap самого себя.

Ниже в качестве модели по умолчанию для разработки используется CodeX.

### 0. Сначала финальный интерфейс

<p align="center">
  <img src="images/0-标题使用.png" alt="Основной интерфейс OpenWorkflows" width="960">
</p>

В основном интерфейсе OpenWorkflows в центре находится blueprint workflows, справа — свойства узлов, снизу — ввод и вывод ИИ.

Интерфейс условно делится на четыре части: история workflows слева, визуальный холст в центре, свойства узлов и частые prompts справа, ввод ИИ и ответы снизу.

### 1. Скачать OpenWorkflows

<p align="center">
  <img src="images/1-下载.png" alt="OpenWorkflows GitHub Releases" width="840">
</p>

Найдите последнюю версию в Releases справа на странице проекта GitHub.

### 2. Сначала настроить большую модель

По умолчанию для запуска используется CLI, уже настроенная в системе. Для настройки можно использовать CC-Switch и похожие инструменты.

### 3. Создать новые workflows и ввести запрос

<p align="center">
  <img src="images/3-新建workflow.png" alt="Создать новые workflows и ввести запрос" width="840">
</p>

После настройки модели нажмите слева "Новые workflows". На холсте появится минимальная структура: Start, один Agent и End.

Рисовать узлы вручную не нужно. Настоящая точка входа — поле ввода ИИ справа внизу. В этом примере я ввел:

```text
Я хочу, чтобы OpenWorkflows поддерживал несколько стилей интерфейса,
по умолчанию использовал дизайн Pencil
и позволял переключаться в Настройки / Внешний вид.
```

После этого можно нажать Ctrl+Enter или кнопку отправки справа внизу. OpenWorkflows превращает этот текст на естественном языке в редактируемый blueprint workflows.

### 4-1. Сгенерировать blueprint workflows

<p align="center">
  <img src="images/4-1生成workflow蓝图.png" alt="Сгенерированный blueprint workflows" width="960">
</p>

После отправки запроса OpenWorkflows сначала преобразует текущий шаг в полноценный workflow.

Blueprint на скриншоте примерно такой:

```text
Start
  -> Параллельно разобрать поддержку внешнего вида
      -> Исследовать существующие входы настроек внешнего вида
      -> Спроектировать систему нескольких стилей
      -> Спроектировать стиль Pencil по умолчанию
  -> Свести план реализации
  -> Реализовать несколько стилей интерфейса
  -> Подключить переключение в Настройки / Внешний вид
  -> Проверить и выполнить регрессионные проверки
  -> Зафиксировать результат поставки
  -> End
```

В свойствах узла справа можно продолжать менять свойства выбранного узла. Но чаще удобнее использовать нижнее поле ввода и просить ИИ изменить узлы blueprint, чтобы итерация продолжалась.

### 4-2. Посмотреть сгенерированный script

<p align="center">
  <img src="images/4-2蓝图脚本.png" alt="Сгенерированный script workflows" width="960">
</p>

Вверху есть пункт "Script". После открытия появляется script, сгенерированный из текущего blueprint.

На скриншоте видно структуры parallel(...) и agent(...). Параллельные узлы превращаются в конкурентно выполняемые ветви, обычные узлы — в отдельные вызовы agent.

Это показывает, что OpenWorkflows не просто рисует блоки. За холстом стоит единая структура workflows, благодаря которой потом можно подключать разные runtimes.

### 5. Продолжить правки с частыми prompts справа

<p align="center">
  <img src="images/5-使用常用提示词.png" alt="Использовать частые prompts для дальнейшего редактирования workflows" width="960">
</p>

После генерации blueprint не обязательно сразу запускать его. Панель "Частые prompts" справа лучше подходит для доработки процесса, хотя можно писать и вручную.

Prompts сгруппированы по сценариям: интерактивное уточнение, ясность, полнота, стоимость, структура, надежность, производительность и параллелизм, проверка и тестирование.

На скриншоте выбран prompt "Уточнить требования". Он заполняет поле ввода ИИ просьбой интерактивно подтвердить ключевые неоднозначности перед изменением blueprint.

Это очень практично. Многие workflows проваливаются не потому, что модель не умеет выполнять задачу, а потому, что цель, границы, пути отказа и стратегия стоимости с самого начала не были ясны.

Есть и другие частые prompts: grill-me, дополнение граничных условий, параллельная оптимизация, единый принцип. Их можно добавлять или изменять самостоятельно.

### 6. Подтвердить границы через интерактивный выбор

<p align="center">
  <img src="images/6-交互选择.png" alt="Интерактивный выбор для подтверждения границ" width="640">
</p>

После нажатия "Уточнить требования" ИИ не меняет граф сразу. Сначала он спрашивает: "В каком объеме должна быть реализована функция переключения стилей интерфейса?"

На скриншоте два варианта: реализовать только стиль Pencil по умолчанию и оставить расширяемую структуру, либо реализовать Pencil и несколько переключаемых стилей.

После выбора ИИ записывает это решение обратно в blueprint workflows и выводит обновленный IRGraph. Этот шаг снижает риск того, что ИИ самовольно уйдет в неверном направлении.

### 7. Нажать Run

<p align="center">
  <img src="images/7-运行.png" alt="Запуск workflows" width="960">
</p>

Когда структура blueprint, конфигурация модели и ключевые границы подтверждены, нажмите "Run" вверху.

Не стоит запускать blueprint сразу после генерации. Сначала проверьте, разумны ли параллельные ветви, находится ли узел сводки после параллельных ветвей, и покрывает ли проверка финальный результат.

Если у какого-то узла просто неясная ответственность, можно сначала изменить его в свойствах узла, а затем запускать.

### 8. Наблюдать состояние выполнения

<p align="center">
  <img src="images/8-运行中.png" alt="Наблюдать состояние выполнения workflows" width="960">
</p>

После запуска верхняя кнопка меняется на "Выполняется... Стоп". Нижний ввод ИИ блокируется, чтобы blueprint не был случайно изменен во время выполнения.

На холсте отображается состояние узлов. На скриншоте Start уже завершен, следующий параллельный узел выполняется, а справа сверху есть счетчик запусков. Если что-то сломается посередине, можно продолжить с предыдущей задачи.

### 9. Переключить стиль интерфейса

<p align="center">
  <img src="images/9-切换风格.png" alt="Переключение стиля интерфейса" width="840">
</p>

После завершения разработки в OpenWorkflows перезапустите программу и переключайте разные стили в Настройки / Внешний вид.

На скриншоте видны карточки стилей Pencil, Deep Night, Aurora, Daylight, Ember и другие. Выбранный стиль влияет на глобальный фон, панели, рамки и цвета состояний выполнения.

### Что я считаю действительно полезным

Главная ценность OpenWorkflows не в том, что он оборачивает prompt в UI.

Он соединяет "запрос -> blueprint -> script -> запуск -> просмотр истории". Можно сначала сгенерировать процесс естественным языком, затем проверить структуру на холсте, при необходимости дополнить границы частыми prompts и только потом запускать.

Одни и те же workflows не обязаны быть привязаны к одной модели. Простые узлы могут использовать дешевые модели, ключевые узлы — более сильные модели, а цель выполнения может дальше расширяться на Claude Code, Codex, Gemini или другие runtimes.

Для сложных задач AI-программирования такое разбиение легче поддерживать, чем один огромный prompt. Если один узел сломался, меняется этот узел. Если ветвь не нужна, она удаляется. Если нужно переиспользовать, можно продолжить из истории.

### Еще рано, но направление стоит внимания

Вся концепция workflows пока довольно ранняя, и OpenWorkflows тоже только начался. Runtime-адаптеры, возможности узлов и script-экосистема будут продолжать меняться.

Но общее направление ясно: AI-программирование не останется надолго в состоянии "открыть чат и вручную продвигать каждый шаг".

Сложные задачи в итоге станут workflows, потому что их можно видеть, редактировать, переносить и переиспользовать.

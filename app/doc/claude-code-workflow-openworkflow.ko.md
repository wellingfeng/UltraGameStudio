# Claude Code에 Dynamic Workflows가 생겼다면, 다른 모델은? 오픈소스 대안 OpenWorkflows

## 최근 Claude Code가 새로 내놓은 dynamic workflows를 보고 있습니다. MCP, Skill, Hooks와 비교하면 이 새 기능을 이야기하는 사람은 아직 많지 않습니다. 아래에서는 그냥 workflows라고 부르겠습니다.

복잡한 작업에서는 예전부터 먼저 조사용 HTML을 만들고, 그다음 기술 방안 HTML로 바꾼 뒤, 마지막에 AI에게 개발을 맡기는 방식이 많이 쓰였습니다. 하지만 결과가 좋지 않은 경우가 많았습니다. 가장 큰 이유는 HTML이 사람이 읽는 텍스트 정보이지 Script가 아니라는 점입니다. 구조화 정보가 부족하고, 순서의 일관성, 병렬화 정도, 경계의 명확성, 작업 분할 방식, 작업 사이의 정보 교환 방식이 모두 불명확해서 AI가 추측해야 하는 부분이 많습니다.

반면 workflows 자체는 Script이기 때문에 이 문제를 근본적으로 해결할 수 있습니다.

또 workflows에는 다각도 탐색, 적대적 검증, 방안 투표 메커니즘도 있습니다. 정확도가 더 높은 이유도 여기에 있습니다. 규모로 이기는 방식입니다. 같은 문제를 5개의 agents가 동시에 실행하고, 다시 하나의 agent가 결과를 모읍니다. 실제로 더 정확해지지만 token도 빠르게 소모됩니다.

이렇게 범용적이라면 왜 특정 모델이나 특정 CLI에 묶여 있어야 할까요?

이 생각을 따라 OpenWorkflows를 개발했습니다. 더 정확히 말하면 AI가 개발했습니다. OpenWorkflows는 Claude Code류 workflows를 시각적 캔버스로 만들고, 같은 흐름을 Claude Code, Codex, Gemini, 그리고 더 많은 로컬 또는 클라우드 런타임에 맞추려 합니다.

이번에는 추상적인 개념을 이야기하지 않고, 바로 스크린샷을 따라가겠습니다. 예제도 구체적입니다. OpenWorkflows가 여러 인터페이스 스타일을 지원하고, 기본값은 Pencil로 사용하며, "설정 / 외관"에서 전환할 수 있게 만드는 것입니다.

개발 과정에서는 가능한 한 OpenWorkflows 안에서 작업해서, 스스로를 부트스트랩할 수 있도록 했습니다.

아래 과정은 CodeX를 개발용 기본 대형 모델로 사용합니다.

<p align="center">

OpenWorkflows의 메인 인터페이스는 중앙에 workflows 블루프린트, 오른쪽에 노드 속성, 하단에 AI 입력과 반환 결과가 있습니다.

메인 인터페이스는 크게 네 부분입니다. 왼쪽은 workflows 히스토리, 중앙은 시각적 캔버스, 오른쪽은 노드 속성과 자주 쓰는 프롬프트, 하단은 AI 입력과 응답입니다.

기본적으로 시스템에 설정된 CLI를 사용해 시작합니다. CC-Switch 같은 도구로 설정할 수 있습니다.

### 3. 새 workflows를 만들고 요구사항 입력

<p align="center">
  <img src="images/3-新建workflow.png" alt="새 workflows를 만들고 요구사항 입력" width="840">
</p>

모델을 설정한 뒤 왼쪽의 "새 workflows"를 클릭합니다. 캔버스에는 Start, 하나의 Agent, End로 구성된 최소 구조가 나타납니다.

실제로 노드를 손으로 그릴 필요는 없습니다. 시작점은 오른쪽 아래의 AI 입력창입니다. 이 예제에서 저는 이렇게 입력했습니다.

```text
OpenWorkflows가 여러 인터페이스 스타일을 지원하고,
기본 디자인은 Pencil을 사용하며,
"설정 / 외관"에서 전환할 수 있으면 좋겠습니다.
```

작성한 뒤 Ctrl+Enter를 누르거나 오른쪽 아래의 전송 버튼을 클릭합니다. OpenWorkflows는 이 자연어를 편집 가능한 workflows 블루프린트로 바꿉니다.

### 4-1. workflows 블루프린트 생성

<p align="center">
  <img src="images/4-1生成workflow蓝图.png" alt="생성된 workflows 블루프린트" width="960">
</p>

요구사항을 보내면 OpenWorkflows는 먼저 현재 단계를 완전한 workflow로 정리합니다.

스크린샷의 블루프린트는 대략 다음과 같습니다.

```text
Start
  -> 외관 지원 방안 병렬 정리
      -> 기존 외관 진입점 조사
      -> 다중 스타일 체계 설계
      -> Pencil 기본 스타일 설계
  -> 구현 방안 요약
  -> 다중 인터페이스 스타일 구현
  -> 설정 / 외관 전환 연결
  -> 검증 및 회귀 확인
  -> 전달 결과 기록
  -> End
```

오른쪽 노드 속성에서는 선택한 노드의 속성을 계속 수정할 수 있습니다. 물론 더 자주 쓰는 방식은 하단 입력창에서 AI에게 블루프린트 노드를 수정하게 하며 계속 반복하는 것입니다.

### 4-2. 생성된 스크립트 보기

<p align="center">
  <img src="images/4-2蓝图脚本.png" alt="생성된 workflows 스크립트" width="960">
</p>

상단에는 "스크립트" 진입점이 있습니다. 클릭하면 현재 블루프린트에서 생성된 스크립트가 나타납니다.

스크린샷에서는 parallel(...)과 agent(...) 같은 구조를 볼 수 있습니다. 병렬 노드는 동시 실행되는 분기가 되고, 일반 노드는 개별 agent 호출이 됩니다.

이것만 봐도 OpenWorkflows가 단순히 그림을 그리는 도구가 아니라는 것을 알 수 있습니다. 캔버스 뒤에는 통일된 workflows 구조가 있고, 그래서 이후에 다른 런타임과 연결할 수 있습니다.

### 5. 오른쪽의 자주 쓰는 프롬프트로 계속 수정

<p align="center">
  <img src="images/5-使用常用提示词.png" alt="자주 쓰는 프롬프트로 workflows 계속 수정" width="960">
</p>

블루프린트가 생성된 뒤 바로 실행할 필요는 없습니다. 오른쪽의 "자주 쓰는 프롬프트"는 흐름을 계속 다듬기에 더 적합합니다. 물론 직접 작성해도 됩니다.

프롬프트는 상호작용식 명확화, 명확성, 완전성, 비용, 구조, 신뢰성, 성능과 병렬성, 검증과 테스트 같은 시나리오별로 묶여 있습니다.

스크린샷에서 선택한 것은 "요구사항 명확화"입니다. 이 프롬프트는 AI가 블루프린트를 수정하기 전에 핵심 모호점을 상호작용으로 확인하도록 AI 입력창에 내용을 채웁니다.

이 설계는 실용적입니다. 많은 workflows가 실패하는 이유는 모델이 못해서가 아니라, 목표, 경계, 실패 경로, 비용 전략이 처음부터 충분히 명확하지 않기 때문입니다.

그 밖에도 grill-me, 경계 조건 보완, 병렬 최적화, 단일 원칙 같은 자주 쓰는 프롬프트가 있습니다. 직접 새로 추가하거나 수정할 수도 있습니다.

### 6. 상호작용 선택으로 경계 확인

<p align="center">
  <img src="images/6-交互选择.png" alt="상호작용 선택으로 경계 확인" width="640">
</p>

"요구사항 명확화"를 누르면 AI는 바로 그래프를 고치지 않고 먼저 "인터페이스 스타일 전환 기능을 어떤 범위까지 구현할까요?"라고 묻습니다.

스크린샷에는 두 가지 선택지가 있습니다. Pencil 기본 스타일만 구현하고 확장 구조를 남길지, 아니면 Pencil과 여러 전환 가능한 스타일을 함께 구현할지입니다.

선택을 마치면 AI는 그 결정을 workflows 블루프린트에 다시 쓰고 업데이트된 IRGraph를 출력합니다. 이 단계는 AI가 스스로 잘못된 방향으로 바꾸는 문제를 줄여 줍니다.

### 7. 실행 클릭

<p align="center">
  <img src="images/7-运行.png" alt="workflows 실행" width="960">
</p>

블루프린트 구조, 모델 설정, 핵심 경계가 모두 확인되면 상단의 "실행"을 클릭합니다.

블루프린트를 생성하자마자 실행하는 것은 권장하지 않습니다. 먼저 병렬 분기가 합리적인지, 요약 노드가 병렬 분기 뒤에 있는지, 검증 노드가 최종 결과를 커버하는지 확인하는 것이 좋습니다.

어떤 노드의 책임만 불분명하다면 먼저 노드 속성에서 수정한 뒤 다시 실행하면 됩니다.

### 8. 실행 상태 관찰

<p align="center">
  <img src="images/8-运行中.png" alt="workflows 실행 상태 관찰" width="960">
</p>

실행 후 상단 버튼은 "실행 중... 중지"로 바뀝니다. 하단 AI 입력은 잠겨서 실행 중에 블루프린트가 흐트러지는 것을 방지합니다.

캔버스에는 노드 상태가 표시됩니다. 스크린샷에서는 Start가 완료되었고, 뒤의 병렬 노드가 실행 중이며, 오른쪽 위에는 실행 카운트도 있습니다. 중간에 실패하면 이전 작업에서 이어서 진행할 수 있습니다.

### 9. 인터페이스 스타일 전환

<p align="center">
  <img src="images/9-切换风格.png" alt="인터페이스 스타일 전환" width="840">
</p>

OpenWorkflows 개발이 완료되면 프로그램을 다시 시작하고 "설정 / 외관"에서 서로 다른 외관 스타일을 전환합니다.

스크린샷에는 Pencil, Deep Night, Aurora, Daylight, Ember 같은 스타일 카드가 보입니다. 스타일을 선택하면 전체 배경, 패널, 테두리, 실행 상태 색상이 바뀝니다.

### 제가 진짜 유용하다고 느끼는 부분

OpenWorkflows의 가장 큰 가치는 prompt를 UI로 감싸는 데 있지 않습니다.

"요구사항 -> 블루프린트 -> 스크립트 -> 실행 -> 히스토리 확인"을 하나로 연결합니다. 자연어로 먼저 흐름을 만들고, 캔버스에서 구조를 확인하고, 필요하면 자주 쓰는 프롬프트로 경계를 보완한 뒤 마지막에 실행할 수 있습니다.

같은 workflows도 반드시 하나의 모델에 묶일 필요가 없습니다. 단순한 노드는 저렴한 모델을, 핵심 노드는 더 강한 모델을 사용할 수 있고, 실행 대상도 Claude Code, Codex, Gemini 또는 다른 런타임으로 계속 확장할 수 있습니다.

복잡한 AI 코딩 작업에서는 이런 분해 방식이 하나의 초장문 prompt보다 유지보수하기 쉽습니다. 어떤 노드가 실패하면 그 노드만 고치면 됩니다. 필요 없는 분기는 삭제하면 됩니다. 재사용하고 싶다면 히스토리에서 이어서 수정하면 됩니다.

### 아직 이르지만 방향은 볼 만합니다

workflows라는 개념 전체가 아직 초기 단계이고, OpenWorkflows 자체도 이제 막 시작했습니다. 런타임 어댑터, 노드 능력, 스크립트 생태계는 계속 변할 것입니다.

하지만 전체 방향은 분명합니다. AI 코딩은 오래도록 "채팅창을 열고 매 단계를 수동으로 밀어붙이는" 방식에 머물지 않을 것입니다.

복잡한 작업은 결국 workflows가 될 것입니다. 볼 수 있고, 편집할 수 있고, 옮길 수 있고, 재사용할 수 있기 때문입니다.

OpenWorkflow

一个受 Unreal Engine 蓝图编辑器启发的可视化工作流编辑器。 用拖拽节点的方式生成与编辑 AI Workflow，并通过一套通用 IR + 多解析器，让同一份工作流可以运行在 Claude Code、Codex、Gemini 等不同大模型之上。

01 概述

Claude Code 最新版本支持以脚本方式编写 Workflow（多智能体编排），表达力强但对非工程用户不友好：需要手写 JS、记忆 agent()/parallel()/pipeline() 等 API，难以一眼看清编排拓扑。

OpenWorkflow 的核心主张：工作流本质是一张有向图。既然是图，就应该像 UE 蓝图那样被看见、拖拽、连线、调试，而脚本只是这张图的一种序列化表示。OpenWorkflow 让图与脚本双向同步，并让脚本能被翻译到不同大模型的运行时。

🎛️ 可视化编辑

蓝图式画布：节点 = 步骤，引脚 = 输入/输出，连线 = 执行流与数据流。

🔁 图↔脚本双向

画布改动实时编译为脚本；导入脚本即可反解析为图。

🧩 多模型通用

统一 IR + 可插拔解析器，一份工作流跑在 Claude / Codex / Gemini。

02 设计目标

✅ 必须达成

  • 蓝图画布：节点增删、连线、拖拽、缩放、对齐网格
  • 图 → 脚本编译（Claude Code workflow 脚本为首发目标）
  • 脚本 → 图解析（导入既有 workflow）
  • 通用 IR：与具体模型/运行时解耦的中间表示
  • 会话管理：新建 Workflow、新建会话、历史记录
  • AI 协作区：与模型对话、由对话驱动图的生成与修改
  • 常用提示词面板：一键追问边界 / 澄清需求

🌟 期望达成

  • 多运行时适配器（Claude Code / Codex CLI / Gemini CLI）
  • 节点级实时执行状态回显（运行中 / 成功 / 失败 / token 消耗）
  • 子工作流（subgraph）与可复用节点库
  • 断点 / 单步 / 重放（参考 UE 蓝图调试）
  • 版本对比与协作
设计原则：图是真相源（Source of Truth），脚本是一种导出格式；但允许工程师在脚本侧编辑后回灌，保持双向一致。

03 系统架构

分层架构，自上而下为：编辑器层、核心模型层（IR）、解析/编译层、运行时适配层。

关键解耦点

  • IR 不绑定任何模型：节点描述「做什么」，适配器决定「在哪跑、怎么跑」。
  • Emitter 与 Parser 成对但独立：可只实现 Emitter（导出）而暂不实现 Parser（导入）。
  • 运行时回流：执行事件（开始/结束/错误/token）以节点 id 为键回写画布。

04 通用 Workflow IR

IR 是 OpenWorkflow 的心脏，一份与模型无关的 JSON 图。下面是一个「评审流水线」工作流的 IR 示例：

// workflow.owf.json — OpenWorkflow IR
{
  "version": "0.1",
  "meta": { "name": "review-changes", "description": "多维度评审并验证" },
  "nodes": [
    { "id":"start",  "type":"start" },
    { "id":"scan",   "type":"agent",
      "params":{ "prompt":"列出变更文件", "model":"haiku", "schema":"FILES" } },
    { "id":"review", "type":"parallel",
      "params":{ "over":"$scan.files", "prompt":"审查该文件" } },
    { "id":"verify", "type":"agent",
      "params":{ "prompt":"对抗式验证每条发现" } },
    { "id":"end",    "type":"end" }
  ],
  "edges": [
    { "from":"start",  "to":"scan",   "kind":"exec" },
    { "from":"scan",   "to":"review", "kind":"exec" },
    { "from":"review", "to":"verify", "kind":"exec" },
    { "from":"verify", "to":"end",    "kind":"exec" }
  ]
}

对应导出的 Claude Code 脚本

export const meta = { name:'review-changes', description:'多维度评审并验证',
  phases:[{title:'Scan'},{title:'Review'},{title:'Verify'}] }

phase('Scan')
const scan = await agent('列出变更文件', { model:'haiku', schema: FILES })
phase('Review')
const reviews = await parallel(scan.files.map(f => () =>
  agent(`审查 ${f}`)))
phase('Verify')
const verdicts = await parallel(reviews.map(r => () =>
  agent(`对抗式验证: ${r}`)))
return { verdicts }

同一份 IR，换一个 Emitter 即可产出 Codex prompt-flow 或 Gemini chain，无需改图。

05 节点类型

引脚约定：▶ exec 执行流 控制顺序； ● data 数据流 传递结果； ◆ io 配置 静态参数。

节点类别引脚映射到脚本
Start / End控制exec out / exec in脚本入口与 return
Agent执行exec ▶▶ · data(prompt,model,schema) → result ●agent(prompt, opts)
Parallel编排exec ▶▶ · over[] ● → results[] ●parallel(items.map(...))
Pipeline编排exec ▶▶ · stages ◆ → out[] ●pipeline(items, s1, s2…)
Phase分组exec ▶▶ · title ◆phase('…') 进度分组
Branch / If控制exec ▶ · cond ● → true▶ / false▶if (...) {}
Loop控制exec ▶ · until ● → body▶ / done▶while (...) {}
Workflow (子图)复用exec ▶▶ · args ● → result ●workflow(name, args)
Log / Note旁路exec ▶▶ · msg ◆log('…')
Variable / Get-Set数据data ● ↔ ●局部 const / 引用 $node.field
节点参数面板支持 schema 选择，对应脚本中的结构化输出（强制 agent 调用 StructuredOutput 工具），保证下游连线拿到的是已校验对象。

◨ 右 · 提示词面板

常用追问清单，每条带三角形箭头 ▷，点击即把该提示词发给 AI，加速澄清需求与边界。

交互要点

  • 蓝图画布：拖拽创建连线、右键菜单加节点、Space 拖拽平移、滚轮缩放、网格吸附。
  • AI 双向：在输入框描述意图 → AI 直接操作画布（增删节点/连线），改动以高亮回显在返回区。
  • 选中节点 → 右侧面板可切换为「节点属性」编辑（prompt / model / schema / isolation）。
  • 底部 AI 返回区与画布共享选区：点节点定位日志，点日志高亮节点。

07 常用提示词面板

面板是「工作流方法论」的固化——把 OpenWorkflow 倡导的澄清式设计变成一键操作。分组可折叠，支持自定义。

清晰度 / Clarity

  • ▷ 当前工作流清晰了吗？
  • ▷ 当前边界清晰了吗？
  • ▷ 每个节点的职责单一吗？

完整性 / Completeness

  • ▷ 是否覆盖了失败 / 超时分支？
  • ▷ 有没有遗漏的并行机会？
  • ▷ 数据从哪来、到哪去明确吗？

正确性 / Correctness

  • ▷ 连线的数据类型匹配吗？
  • ▷ 是否存在环 / 死锁？
  • ▷ 结构化输出 schema 定义了吗？

成本 / Cost

  • ▷ 模型档位（haiku/sonnet/opus）合理吗？
  • ▷ 并发上限与预算控制了吗？
  • ▷ 哪些节点可降级？

点击三角形 ▷ → 该提示词作为一条消息发给 AI，AI 据此审视当前画布并给出修改建议或直接落地。

08 通用解析器

解析器把不同来源/目标的工作流表示与 IR 互转。采用注册表 + 适配器模式，新增模型只需实现一对 parse/emit。

interface WorkflowAdapter {
  id: string;              // 'claude-code' | 'codex' | 'gemini'
  parse(src: string): IRGraph;   // 外部表示 → IR
  emit(ir: IRGraph): string;    // IR → 外部表示
  capabilities(): CapSet;     // 支持哪些节点/特性
}

registry.register(ClaudeCodeAdapter);  // 首发
registry.register(CodexAdapter);
registry.register(GeminiAdapter);

解析策略

  • Claude Code 脚本：用 AST（acorn/babel）解析，识别 agent/parallel/pipeline/phase/workflow 调用与数据依赖，重建节点与边。
  • 声明式 (YAML/JSON)：直接 schema 映射，最稳妥，作为跨模型的「无损」交换格式。
  • 能力协商：emit 前用 capabilities() 检查目标运行时是否支持某节点（如 worktree 隔离），不支持则降级或告警。
脚本是图灵完备的，存在无法静态还原为图的写法。策略：可还原的子集映射为节点；无法还原的片段封装为 CodeBlock 节点（黑盒），保证「往返不丢」。

09 数据模型

Session 会话

{
  id, title, createdAt,
  workflowId,
  messages: Message[],   // AI 对话
  cursor: NodeId          // 当前焦点节点
}

Workflow 工作流

{
  id, name, version,
  ir: IRGraph,            // 真相源
  target: 'claude-code',  // 默认运行时
  layout: {nodeId:{x,y}}  // 画布坐标
}

Node 节点

{
  id, type, label,
  params: {...},
  ports: { in:[], out:[] },
  runtime?: {status, tokens, log}
}

Edge 连线

{
  id, from:{node,port},
  to:{node,port},
  kind: 'exec' | 'data'
}

持久化：会话与历史落地为 ~/.openworkflow/sessions/*.json；工作流为 *.owf.json，可导出为目标脚本文件。
